Git Basics: The Foundation

Installing Git

# Windows - Download from git-scm.com or use:
winget install --id Git.Git -e --source winget

# macOS
brew install git

# Linux (Ubuntu/Debian)
sudo apt install git

# Verify installation
git --version

First-Time Configuration

# Set your identity (used in every commit)
git config --global user.name "Dharmik Gohil"
git config --global user.email "your-email@example.com"

# Set default branch name
git config --global init.defaultBranch main

# Set VS Code as default editor
git config --global core.editor "code --wait"

# Enable colored output
git config --global color.ui auto

# View all configurations
git config --list

Repository Operations

# Initialize a new repository
git init

# Clone an existing repository
git clone https://github.com/dharmikgohil013/portfolio.git

# Clone with a specific branch
git clone -b develop https://github.com/user/repo.git

# Clone into a specific folder
git clone https://github.com/user/repo.git my-folder

The Daily Workflow — Dharmik Gohil's Git Routine

Here's the Git workflow I, Dharmik Gohil, follow for every project:

# 1. Check current status (do this OFTEN)
git status

# 2. See what changed
git diff                    # Unstaged changes
git diff --staged           # Staged changes
git diff HEAD~1             # Compare with last commit

# 3. Stage files for commit
git add filename.js         # Stage specific file
git add src/                # Stage entire folder
git add .                   # Stage everything (use carefully!)
git add -p                  # Interactive staging (pick hunks)

# 4. Commit with a meaningful message
git commit -m "feat: add player movement system"
git commit -am "fix: resolve collision detection bug"  # Stage + Commit

# 5. Push to remote
git push origin main
git push origin feature/player-combat
💡 Dharmik's Commit Message Convention: I follow the Conventional Commits standard: type: description. Types include: feat (new feature), fix (bug fix), docs (documentation), style (formatting), refactor, test, and chore (maintenance).

Branching — The Power of Parallel Development

Branching is Git's killer feature. It lets you work on features without breaking the main codebase:

# List all branches
git branch              # Local branches
git branch -r           # Remote branches
git branch -a           # All branches

# Create a new branch
git branch feature/inventory-system

# Switch to a branch
git checkout feature/inventory-system
# Or the modern way:
git switch feature/inventory-system

# Create and switch in one command
git checkout -b feature/inventory-system
git switch -c feature/inventory-system

# Rename a branch
git branch -m old-name new-name

# Delete a branch (local)
git branch -d feature/old-feature        # Safe delete
git branch -D feature/old-feature        # Force delete

# Delete a remote branch
git push origin --delete feature/old-feature

Merging & Rebasing

Merge (Preserves History)

# Merge a feature branch into main
git checkout main
git merge feature/inventory-system

# Merge with a commit message
git merge feature/inventory-system -m "Merge inventory system feature"

# Abort a merge (if conflicts are too complex)
git merge --abort

Rebase (Linear History)

# Rebase feature branch onto latest main
git checkout feature/inventory-system
git rebase main

# Interactive rebase (squash, reorder, edit commits)
git rebase -i HEAD~5     # Last 5 commits
git rebase -i main       # All commits since branching from main

# Abort a rebase
git rebase --abort

# Continue after resolving conflicts
git rebase --continue
⚠️ Dharmik's Warning: Never rebase commits that have been pushed to a shared branch. Rebase rewrites history — if others have based work on those commits, it will cause serious conflicts. Rebase is for YOUR local branches only.

Resolving Merge Conflicts

Conflicts happen when two branches modify the same code. Here's how Dharmik Gohil handles them:

# When a conflict occurs, Git marks the file:
<<<<<<< HEAD
// Your changes (current branch)
float moveSpeed = 5f;
=======
// Their changes (incoming branch)
float moveSpeed = 8f;
>>>>>>> feature/speed-boost

# Resolution steps:
# 1. Open the conflicted file
# 2. Choose which code to keep (or combine both)
# 3. Remove the conflict markers (<<<, ===, >>>)
# 4. Stage the resolved file
git add resolved-file.cs
# 5. Continue the merge/rebase
git commit  # For merge
git rebase --continue  # For rebase
💡 Pro Tip from Dharmik: VS Code has an excellent built-in merge editor. When you open a conflicted file, it shows "Accept Current", "Accept Incoming", "Accept Both", and "Compare Changes" buttons above each conflict. Use them!

Stashing — Save Work Without Committing

# Stash current changes (saves and reverts working directory)
git stash
git stash save "WIP: inventory UI changes"

# List all stashes
git stash list

# Apply the latest stash
git stash pop             # Apply and remove from stash
git stash apply           # Apply but keep in stash

# Apply a specific stash
git stash pop stash@{2}

# View stash contents
git stash show -p stash@{0}

# Drop a stash
git stash drop stash@{0}

# Clear all stashes
git stash clear

Viewing History & Logs

# Basic log
git log

# One-line log (Dharmik's favorite)
git log --oneline

# Graph view (visualize branches)
git log --oneline --graph --all --decorate

# Log with file changes
git log --stat

# Log for specific file
git log -- path/to/file.cs

# Search commits by message
git log --grep="inventory"

# Search commits by author
git log --author="Dharmik"

# Show changes in a specific commit
git show abc1234

# Show who changed each line (blame)
git blame path/to/file.cs

Undoing Changes — The Safety Net

# Discard changes in working directory
git checkout -- filename.cs
git restore filename.cs          # Modern way

# Unstage a file
git reset HEAD filename.cs
git restore --staged filename.cs # Modern way

# Undo the last commit (keep changes)
git reset --soft HEAD~1

# Undo the last commit (discard changes) ⚠️
git reset --hard HEAD~1

# Create a new commit that undoes a previous commit
git revert abc1234

# Recover deleted branch/commit (within 30 days)
git reflog
git checkout -b recovery-branch abc1234
⚠️ Danger Zone: git reset --hard permanently deletes uncommitted changes. Use git stash first if you might need those changes later. Dharmik Gohil has lost code to careless hard resets — learn from my mistakes!

GitHub — Collaboration Features

Pull Requests (PRs)

Pull requests are the backbone of team collaboration on GitHub. Here's my workflow:

  1. Create a feature branch: git checkout -b feature/new-weapon
  2. Make your changes and commit them.
  3. Push to GitHub: git push origin feature/new-weapon
  4. Open a Pull Request on GitHub (compare your branch to main).
  5. Request review from teammates.
  6. Address feedback and push more commits.
  7. Merge the PR once approved.

Forking & Contributing to Open Source

# 1. Fork the repository on GitHub (click "Fork" button)

# 2. Clone YOUR fork
git clone https://github.com/YOUR-USERNAME/repo.git

# 3. Add the original repo as "upstream"
git remote add upstream https://github.com/ORIGINAL-OWNER/repo.git

# 4. Keep your fork updated
git fetch upstream
git checkout main
git merge upstream/main

# 5. Create a feature branch, make changes, push to YOUR fork
git checkout -b fix/typo-in-readme
git push origin fix/typo-in-readme

# 6. Open a PR from YOUR fork to the original repo

GitHub Actions — CI/CD Basics

# .github/workflows/ci.yml
name: CI Pipeline
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Install dependencies
        run: npm ci
      - name: Run tests
        run: npm test
      - name: Build
        run: npm run build

Advanced Git Tips from Dharmik Gohil

  • Cherry-pick: Apply a specific commit to another branch: git cherry-pick abc1234
  • Bisect: Find the exact commit that introduced a bug: git bisect start → git bisect bad → git bisect good v1.0
  • Worktrees: Check out multiple branches simultaneously: git worktree add ../feature-branch feature/new-ui
  • Submodules: Include other repos inside yours: git submodule add https://github.com/user/library.git
  • Tags: Mark release versions: git tag -a v1.0.0 -m "First release"
  • Aliases: Create shortcuts: git config --global alias.co checkout
